Creación de procesos

Concurrencia en Elixir

  • Elixir es un lenguaje funcional concurrente.

  • La concurrencia es una de las principales virtudes de la máquina virtual BEAM, en la que se ejecutan los programas de Elixir (y Erlang).

  • El modelo de concurrencia de Elixir está basado en actores y paso de mensajes.

    • Esto simplifica notablemente el razonamiento sobre programas concurrentes, ya que no hay memoria compartida.

Concurrencia basada en actores

  • Un actor es un proceso independiente que no comparte memoria con otros procesos.

  • La máquina virtual permite crear actores, que se envían mensajes entre ellos.

  • Un proceso de Elixir no es un proceso del sistema operativo ni una hebra de ejecución.

    • Los procesos de Elixir son ligeros: es posible tener en ejecución cientos de miles de procesos.

Creación de un proceso: spawn/3

spawn(Modulo, :funcion, [arg1, ..., argn])

  • La función spawn/3 crea un nuevo proceso.

  • El proceso recién creado ejecutará la función del Modulo indicado, pasando los argumentos indicados en el tercer parámetro.

Ejemplo

defmodule Procesos do
  def escribir_mensaje() do
    IO.puts("Hola, mundo!")
  end
end
> spawn(Procesos, :escribir_mensaje, [])
Hola, mundo!
#PID<0.136.0>

Process identifiers (PIDs)

  • Cada proceso tiene un identificador (PID), que es de la forma <x.y.z>, donde x, y, z son números.

  • El PID de un proceso servirá para enviar mensajes a ese proceso, u obtener información del mismo.

  • La función spawn devuelve el PID del proceso creado.

> p = spawn(Procesos, :escribir_mensaje, [])
Hola, mundo!
...

> p
#PID<0.136.0>    PID del proceso

Finalización de un proceso

  • Cuando la función que está ejecutando el proceso finaliza, el proceso muere.
> p = spawn(Procesos, :escribir_mensaje, [])
Hola, mundo!
#PID<0.136.0>

> Process.alive?(p)
false
  • Supongamos que modificamos la función escribir_mensaje para que espere 10 segundos antes de terminar.

  IO.puts("Hola, mundo!")
  Process.sleep(10000)   `sleep/1` recibe una cantidad de tiempo (en ms)

Envío y recepción de mensajes

Enviar mensajes: send/2

send(pid, mensaje)

  • Envía el mensaje dado al proceso con el pid dado.

  • El mensaje puede ser cualquier término de Elixir: enteros, átomos, listas, diccionarios, funciones, etc.

  • Si el proceso con el pid indicado estaba muerto, la función send no hace nada.

  • La función send devuelve el mensaje enviado

    • No se suele utilizar el valor de retorno de send.

Buzón de mensajes

  • Cada proceso tiene un buzón de mensajes, al que llegan todos los mensajes que se le envían.

  • Este proceso se comporta mediante una cola de mensajes.

    • El primer mensaje recibido se coloca al principio de la cola, el siguiente después, etc.

Ejemplo

> self()
#PID<0.134.0>

> pid = spawn(...)
#PID<0.208.0>

> send(pid, :hola)
:hola

  • El mensaje :hola queda almacenado en la cola de mensajes del proceso destinatario.

> Process.info(p, :messages)
{:messages, [:hola]}

Recibir mensajes: receive

receive do
  patron1 -> expr1
  patron2 -> expr2
  ...
  patronn -> exprn
end

  • Consulta el primer mensaje de la cola de mensajes, e intenta ajustarla con alguno de los patrones del receive.

  • Si patroni es el primer patrón que encaja con el mensaje, se evaluará la expresión expri, y su resultado será el resultado del receive.

    • En ese caso, el mensaje se elimina de la cola de mensajes.

Caso especial 1

La cola está vacía

  • Si el proceso que ejecuta el receive no tiene ningún mensaje en la cola de mensajes, la ejecución del proceso queda suspendida en el receive hasta que llegue algún mensaje.
def espera_receive() do
  IO.puts("Esperando mensaje...")
  receive do
    :hola -> IO.puts("Mensaje recibido")
  end
  IO.puts("Finalizando proceso...")
end
> p = spawn(Procesos, :espera_receive, [])
Esperando mensaje...
#PID<0.226.0>

> Process.alive?(p)
true

> send(p, :hola)
Mensaje recibido
Finalizando proceso...
:hola

> Process.alive?(p)
false

Caso especial 2

El primer mensaje no ajusta con ningún patrón del receive

  • Si el primer mensaje de la cola de mensajes no ajusta con ninguno de los patrones de las cláusulas:

    • El primer mensaje se extrae de la cola y se guarda al principio de una segunda cola (save queue).

  • Se toma el siguiente mensaje de la cola de mensajes, y se intenta ajustar con alguno de los patrones.

    • Si ajusta, se evalúa la expresión correspondiente.

    • Si no ajusta, se extrae de la cola y se guarda en la save queue. Se prueba con el siguiente mensaje, y así sucesivamente.

  • Cuando finalmente hay un mensaje que sí ajusta con alguno de los patrones del receive, todos los mensajes de la save queue se vuelven a colocar en la cola de mensajes «principal».

    • …para que puedan ser procesados por un receive posterior.

    • Los mensajes se extraen, uno a uno, de la save queue, y se insertan al principio de la cola de mensajes.

      • De este modo, siguen manteniendo el orden que tenían originalmente en la cola de mensajes.

Variables en los patrones

  • Los patrones de un receive pueden tener variables, que quedarán ligadas a las partes del mensaje enviado que ajusten con ellas.
def calculadora() do
  receive do
    {:suma, x, y} ->
      IO.puts("#{x} + #{y} = #{x + y}")
    {:resta, x, y} ->
      IO.puts("#{x} - #{y} = #{x - y}")
  end
end
> p = spawn(Procesos, 
            :calculadora, [])
#PID<0.167.0>

> send(p, {:suma, 3, 4})
3 + 4 = 7
{:suma, 3, 4}

Cláusulas catch-all

  • Al igual que en un case, si un patrón es una variable, ajustará con cualquier mensaje que no haya ajustado con los patrones anteriores:
def calculadora() do
  receive do
    {:suma, x, y} ->
      IO.puts("#{x} + #{y} = #{x + y}")
    {:resta, x, y} ->
      IO.puts("#{x} - #{y} = #{x - y}")
    otro ->    Cláusula catch-all 
      IO.puts("Mensaje desconocido")
      IO.inspect(otro)
  end
end

{:mult, 1, 4}

Guardas en un receive

  • Se permiten guardas en las cláusulas de un receive.
    • Para ello se utiliza when, al igual que en los case.
  • Si un mensaje encaja con un patrón, pero no satisface su guarda, será como si no hubiese encajado con el patrón.
def calculadora() do
  receive do
    {:suma, x, y} when x >= 0 ->
      IO.puts("#{x} + #{y} = #{x + y}")
    {:resta, x, y} ->
      IO.puts("#{x} - #{y} = #{x - y}")
    otro ->
      IO.puts("Mensaje desconocido")
      IO.inspect(otro)
  end
end
> p = spawn(Procesos, 
            :calculadora, [])
#PID<0.169.0>

> send(p, {:suma, -4, 0})
Mensaje desconocido
{:suma, -4, 0}

La cláusula after

receive do
  patron1 -> expr1
  patron2 -> expr2
  ...
  patronn -> exprn
  after num -> exprafter
end
  • num es un número entero que indica una cantidad de tiempo en milisegundos.

  • Si el proceso que ejecuta el receive tiene mensajes en la cola que ajuste con alguno de los patrones, se ejecuta la cláusula correspondiente, como si no hubiera un after.

  • Pero si ninguno de los mensajes de la cola encaja con alguno de los patrones, del receive, la ejecución se suspende hasta que:

    • llegue algún mensaje que ajuste con algún patrón, o bien,
    • hayan transcurrido num milisegundos.

  • En este segundo caso, se evaluará la expresión exprafter.

Ejemplo

> receive do
    x -> IO.puts("Mensaje recibido!")
    after 2000 -> IO.puts("Demasiado tarde!")
  end

...Transcurren dos segundos...
Demasiado tarde!
> send(self(), :ok)
> receive do
    x -> IO.puts("Mensaje recibido!")
    after 2000 -> IO.puts("Demasiado tarde!")
  end

Mensaje recibido!

Casos especiales en cláusula after

  • after 0

    • Sirve para no bloquear la ejecución si no hay mensajes en la cola que ajusten con los patrones del receive.

    • Si la cola esta vacía, o ninguno de sus mensajes ajusta con ningún patrón del receive, se evaluará la expresión del after de manera inmediata.

  • after :infinity

    Equivale a no colocar un after.

Bucle de proceso

Ejemplo

def calculadora() do
  receive do
    {:suma, x, y} ->
      IO.puts("#{x} + #{y} = #{x + y}")
    {:resta, x, y} ->
      IO.puts("#{x} - #{y} = #{x - y}")
  end
end
> p = spawn(Procesos, 
            :calculadora, [])
#PID<0.169.0>

> send(p, {:suma, 1, 3})
1 + 3 = 4
{:suma, 1, 3}

> send(p, {:resta, 0, 4})
...No ocurre nada...
  • ¿Por qué no se imprime nada tras enviar el mensaje {:resta, 0, 4}?

  • Cuando el proceso ha recibido el primer mensaje {:suma, 1, 3}, se ejecuta el IO.puts correspondiente a la guarda {:suma, x, y} y la función calculadora finaliza.

    • Por tanto, el proceso muere.

  • Cuando enviamos el mensaje {:resta, 0, 4}, ya no existe ningún proceso con el PID p.

    • Por tanto, la llamada a send(p, {:resta, 0, 4}) no hace nada.

¿Y si quisiéramos que calculadora pudiese recibir más de un mensaje?

Bucle de recepción de mensajes

  • Podemos hacer que, tras procesar un mensaje, calculadora se haga una llamada recursiva a sí misma para volver a esperar el siguiente mensaje:
def calculadora() do
  receive do
    {:suma, x, y} ->
      IO.puts("#{x} + #{y} = #{x + y}")
      calculadora()
    {:resta, x, y} ->
      IO.puts("#{x} - #{y} = #{x - y}")
      calculadora()
  end
end
  • De este modo, la llamada recursiva ejecutará otro receive y esperará al siguiente mensaje.
> p = spawn(Procesos, :calculadora, [])
#PID<0.216.0>

> send(p, {:suma, 4, 9})
4 + 9 = 13
{:suma, 4, 9}

> send(p, {:resta, 10, 3})
10 - 3 = 7
{:resta, 10, 3}

> send(p, {:suma, 1, 1})
1 + 1 = 2
{:suma, 1, 1}

Sobre el desbordamiento de pila

def calculadora() do
  receive do
    {:suma, x, y} ->
      IO.puts("#{x} + #{y} = #{x + y}")
      calculadora()
    {:resta, x, y} ->
      IO.puts("#{x} - #{y} = #{x - y}")

end

  • ¿Puede haber desbordamiento de la pila de llamadas si calculadora se hace demasiadas llamadas a sí misma?

  • No, porque la función calculadora tiene recursión de cola.

  • Por tanto, el proceso que ejecuta calculadora puede estar ejecutándose de manera permanente, y nunca terminará.

Terminación de un proceso

¿Y si queremos que el proceso calculadora finalice?
Hay dos maneras:

  1. Añadir una cláusula al receive que no haga una llamada recursiva a calculadora:
def calculadora() do
  receive do
    {:suma, x, y} ->
      IO.puts("#{x} + #{y} = #{x + y}")
      calculadora()
    {:resta, x, y} ->
      IO.puts("#{x} - #{y} = #{x - y}")
      calculadora()
    :salir -> :ok     No se hace llamada recursiva 
  end
end
> p = spawn(Procesos, :calculadora, [])
...

> send(p, {:suma, 3, 5})
3 + 5 = 8
{:suma, 3, 5}

> send(p, :salir)
:salir

> Process.alive?(p)
false

  1. Utilizar Process.exit/2:

    > p = spawn(Procesos, :calculadora, [])
...


> Process.exit(p, :kill)

    El segundo parámetro de exit es el motivo de la finalización del proceso.

    El átomo :kill fuerza la finalización.

Modelo cliente-servidor

Modelo cliente-servidor

  • En la arquitectura cliente-servidor, un servidor es un proceso que proporciona un determinado servicio a uno o varios clientes.

    • El servidor actúa de forma pasiva, esperando a recibir peticiones de los clientes.
    • Las peticiones y respuestas se realizan mediante mensajes.

Llamadas síncronas vs asíncronas

Llamada síncrona

Cuando un cliente envía una petición a un servidor, el cliente espera hasta recibir una respuesta del mismo.

Llamada asíncrona

Cuando un cliente envía una petición a un servidor, el cliente continúa su ejecución sin esperar la respuesta del servidor.

Ejemplo: calculadora

  • Partimos de la siguiente implementación del proceso calculadora:

    def calculadora() do
  receive do
    {:suma, x, y} ->
      IO.puts("#{x} + #{y} = #{x + y}")
      calculadora()
    {:resta, x, y} ->
      IO.puts("#{x} - #{y} = #{x - y}")
      calculadora()
    :salir ->
      IO.puts("Adios")
    _ ->
      IO.puts("Mensaje desconocido")
      calculadora()        
  end
end

  • En lugar de imprimir el resultado de la operación por pantalla, queremos responder al cliente con dicho resultado.

    def calculadora() do
  receive do
    {:suma, x, y} ->
      send(???, x + y)
      calculadora()
    {:resta, x, y} ->
      send(???, x - y)
      calculadora()
    # ...
  end
end

🤔 ¿Cómo obtenemos el PID del cliente que envió la petición?

  • Solución: hacer el cliente envíe su propio PID junto con la petición.

  • Todas las peticiones tendrán la forma {pid, mensaje}, donde mensaje es el contenido de la petición (payload).
def calculadora() do
  receive do
    {pid, {:suma, x, y}} ->
      send(pid, x + y)
      calculadora()
    {pid, {:resta, x, y}} ->
      send(pid, x - y)
      calculadora()
    {_, :salir} -> :ok
    _ ->
      IO.puts("error")
      calculadora()
  end
end
> p = spawn(ClienteServidor, 
            :calculadora, [])
#PID<0.125.0>

     El cliente es la shell 
> send(p, {self(), {:suma, 1, 10}})
{#PID<0.118.0>, {:suma, 1, 11}}

> receive do
    result -> result
  end

Funciones envoltorio

  • Podemos encapsular la lógica del cliente en sendas funciones suma/3 y resta/3, que hagan el envío y la recepción de los mensajes correspondientes.

  • Estas funciones reciben como primer parámetro el PID del servidor.

    def suma(pid, x, y) do
  send(pid, {self(), {:suma, x, y}})
  receive do: (result -> result)
end

def resta(pid, x, y) do
  send(pid, {self(), {:resta, x, y}})
  receive do: (result -> result)
end

Problema: recepción de respuesta

  • Hemos visto que, cuando un cliente envía una petición mediante send, hace un receive para esperar una respuesta del servidor.

🤔 ¿Qué pasa si el cliente ha recibido previamente un mensaje de un tercer proceso?

> p = spawn(ClienteServidor, :calculadora, [])
#PID<0.128.0>

> send(self(), :mensaje_inoportuno) El cliente se envía un mensaje a sí mismo 

> ClienteServidor.suma(p, 4, 5)
:mensaje_inoportuno  😲 

> ClienteServidor.resta(p, 10, 3)
9   😰

  • El receive de suma obtiene el mensaje «intruso» en el receive, porque es el primero en la cola de mensajes.

  • El resultado real de la operación suma (9) se queda en la cola de mensajes del cliente, y es el que se obtiene en la siguiente llamada a resta.

¿Cómo evitarlo?

  • Cuando un cliente hace un receive tras enviar la petición al servidor, debería exigir que la respuesta recibida sea la del servidor, y no un mensaje cualquiera de otro proceso.

  • Para asegurar esto, hacemos que el servidor también devuelva su propio PID junto con la respuesta.

  • La implementación del código del servidor queda del siguiente modo:

    def calculadora() do
  receive do
    {pid, {:suma, x, y}} ->
      send(pid, {self(), x + y})   self() = PID del servidor 
      calculadora()
    {pid, {:resta, x, y}} ->
      send(pid, {self(), x - y})
      calculadora()
    {_, :salir} -> :ok
    _ ->
      IO.puts("Mensaje desconocido")
      calculadora()
  end
end

  • A partir de ahora, el cliente exigirá que la respuesta recibida provenga del servidor:

    def suma(pid, x, y) do
  send(pid, {self(), {:suma, x, y}})
  receive do
    {^pid, result} -> result
  end
end

def resta(pid, x, y) do
  send(pid, {self(), {:resta, x, y}})
  receive do
    {^pid, result} -> result
  end
end

  • Utilizando el operador pin (^), estamos exigiendo que la primera componente del mensaje recibido sea igual que el parámetro pid.

> p = spawn(ClienteServidor, :calculadora, [])
#PID<0.128.0>

> send(self(), :mensaje_inoportuno)

> ClienteServidor.suma(p, 4, 5)
9

> ClienteServidor.resta(p, 10, 3)
7

Refactorización de código

  • Para evitar duplicidad de código en las funciones suma y resta, definimos una función call que agrupa las partes comunes:
defp call(pid, msg) do
  send(pid, {self(), msg})
  receive do
    {^pid, result} -> result
  end
end

def suma(pid, x, y), do: call(pid, {:suma, x, y})
def resta(pid, x, y), do: call(pid, {:resta, x, y})
  • Del mismo modo, suele ser habitual abstraer el inicio del servidor en una función start:
def start() do
  spawn(ClienteServidor, :calculadora, [])
end
> p = ClienteServidor.start()

> ClienteServidor.suma(p, 4, 1)
5

Uso de referencias

  • El enfoque anterior es suficiente cuando el cliente solamente recibe llamadas síncronas.

  • Sin embargo, si se mezclan llamadas síncronas y asíncronas, es posible que el cliente, cuando espera la respuesta de una petición al servidor, reciba un mensaje del servidor correspondiente a otra petición distinta.

  • Para evitar esto, se utilizan referencias (references).

  • Las referencias son un tipo básico de Elixir que actúan como identificadores únicos.

  • Cada vez que el cliente hace una petición al servidor, envía una referencia nueva que identifica unívocamente a esa petición, y espera que el servidor devuelva esa misma referencia en su respuesta.

  • Más información: make_ref/0.

Servidores con estado

Servidores con estado

  • El servidor calculadora explicado anteriormente era un servidor sin estado (stateless).

    • Cada petición era completamente independiente de la anterior.

  • No obstante, en algunos casos hemos de almacenar información que se conserve entre peticiones.

    • Esta información conforma el estado del servidor.

Ejemplo: servidor de tickets

  • Supongamos que queremos implementar un servidor que expida tickets numerados desde 1 hasta un número N.

  • Cuando el servidor recibe una petición :get_ticket, devuelve una respuesta con el número de ticket más bajo que aún no haya sido asignado.

  • Cuando el servidor ha respondido con último ticket (N), finaliza.

Ejemplo de funcionamiento

> p = ServidorTickets.start(5)   Repartimos 5 tickets 
> ServidorTickets.get_ticket()
{:ticket, 1}
> ServidorTickets.get_ticket()
{:ticket, 2}

...

> ServidorTickets.get_ticket()
{:ticket, 5}
10:52:20.380 [info] Servidor de tickets finaliza

Implementación

  • Comenzamos con la función recursiva de cola que se encarga de recibir y atender los mensajes en el servidor.

  • Esta función tiene dos parámetros:

    • El número del siguiente ticket a expedir.
    • El número de tickets total.

  • Estos dos parámetros conforman el estado del servidor.

def tickets_loop(siguiente_ticket, max_tickets) do
  receive do
    {pid, :get_ticket} ->
      send(pid, {self(), {:ticket, siguiente_ticket}})
      tickets_loop(siguiente_ticket + 1, max_tickets)    Siguiente estado 

    _ ->
      Logger.warning("Mensaje incorrecto")
      tickets_loop(siguiente_ticket, max_tickets)
  end
end

Añadimos el caso base:

def tickets_loop(siguiente_ticket, max_tickets) 
    when siguiente_ticket > max_tickets do
  Logger.info("Servidor de tickets finaliza")

Funciones útiles sobre procesos

Registro de nombres

  • Habitualmente se suele utilizar un PID para enviar un mensaje a un proceso.

  • Esto nos obliga a propagar el PID de un proceso X a todos los procesos que hagan uso de X.

  • A veces es más fácil utilizar un átomo para hacer referencia a un proceso. Por ejemplo:

    send(:servidor_tickets, {self(), :get_ticket})

Función Process.register/2

  • Elixir dispone de un registro que asocia nombres de procesos (átomos) con PIDs.

  • La función Process.register/2 añade una nueva entrada a ese registro.

    > p = spawn(...)
> Process.register(p, :mi_nombre)

  • A partir de ese momento, se puede utilizar el nombre asignado al proceso en la función send/2:

    > send(:mi_nombre, ...)

Ejemplo: servidor de tickets

  • Para facilitar el uso del servidor de tickets, le asignamos un nombre.

    • Esto solo es útil cuando se tiene un único proceso del servidor de tickets en ejecución.

  • Por convenio, suele utilizarse el nombre del módulo en el que se define el servidor.

    def start(num_tickets) do
  p = spawn(ServidorTickets, :tickets_loop, [1, num_tickets])
  Process.register(p, ServidorTickets)     Nombre asignado al proceso 
end

def get_ticket(), do: call(ServidorTickets, :get_ticket)
